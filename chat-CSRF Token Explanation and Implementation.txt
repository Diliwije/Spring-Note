# Comprehensive Notes: CSRF (Cross-Site Request Forgery) in Spring Boot

## 1. Theory: What is CSRF?

**CSRF** stands for **Cross-Site Request Forgery**. It is an attack that forces an end user to execute unwanted actions on a web application in which they're currently authenticated.

### The Problem Scenario
1.  **Authentication:** A user logs into a secure application (e.g., a banking site or your Spring Boot app). The server issues a **Session ID** (cookie) to keep the user logged in.
2.  **The Vulnerability:** The user visits a malicious website in another tab while still logged into the secure application.
3.  **The Attack:** The malicious website contains a script that automatically sends a request (e.g., "Transfer Money" or "Delete Account") to the secure application's server.
4.  **The Result:** Because the browser automatically attaches the valid **Session ID** cookie to the request, the server thinks the request is legitimate and processes it.

### The Solution: CSRF Token
To prevent this, Spring Security implements a **CSRF Token**.
*   **State-Changing Requests:** Methods like `POST`, `PUT`, `DELETE` (which modify data) require a unique, unpredictable token.
*   **Read-Only Requests:** Methods like `GET` generally do not require a CSRF token.
*   **Mechanism:** The server generates a token. The client must send this token back with every state-changing request. If the token is missing or invalid, the server rejects the request.

---

## 2. Spring Security Default Behavior

*   **CSRF Protection:** Enabled by default in Spring Security for session-based authentication.
*   **GET Requests:** Allowed without a CSRF token.
*   **POST/PUT/DELETE Requests:** Blocked (HTTP 403 Forbidden) if the CSRF token is not present in the request header or parameter.

### C. The CSRF Token Fetcher (CsrfTokenController.java)
Since we are testing with Postman (not a browser form), we need an endpoint to retrieve the current CSRF token to use in our headers.

```java
package com.example.demo;

import org.springframework.security.web.csrf.CsrfToken;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;
import jakarta.servlet.http.HttpServletRequest; // Or javax.servlet.http depending on version

@RestController
public class CsrfTokenController {

    // Endpoint to fetch the CSRF Token
    @GetMapping("/csrf-token")
    public CsrfToken getCsrfToken(HttpServletRequest request) {
        // Spring Security stores the token in the request attribute "_csrf"
        return (CsrfToken) request.getAttribute("_csrf");
    }
}
```

## 5. Alternative Solutions & Configuration

The video mentions that while CSRF tokens are standard for browser-based sessions, there are other ways to handle security depending on the architecture.

### A. Disabling CSRF (For Stateless APIs)
If you are building a REST API that uses **JWT (JSON Web Tokens)** or is purely stateless, you often disable CSRF protection because you aren't relying on session cookies that browsers auto-submit.

**Configuration Example (SecurityConfig.java):**
```java
@Bean
public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    http
        .csrf(csrf -> csrf.disable()) // Disables CSRF protection
        .authorizeHttpRequests(auth -> auth.anyRequest().authenticated());
    return http.build();
}
```
*   **Warning:** Do not disable CSRF if you are using standard Session/Cookie authentication in a browser.

### B. Stateless Sessions
Instead of a server-side session ID, use a stateless token (like JWT). Since there is no session cookie automatically sent by the browser, the CSRF attack vector is removed.

### C. SameSite Cookie Attribute
Configure cookies to have the `SameSite=Strict` attribute. This tells the browser not to send cookies if the request originates from a different website (cross-site).

## 7. Key Takeaways from Video
1.  **Session ID vs. CSRF Token:** The Session ID proves *who* you are. The CSRF Token proves *you* intended to send *this specific request*.
2.  **Browser vs. Postman:** Browsers handle CSRF tokens automatically in forms (via hidden input fields). Tools like Postman require you to manually fetch the token and add it to the Headers.
3.  **Debugging:** If you get a 403 Error on a POST request despite having correct Login credentials, check if you are missing the CSRF Token.
4.  **Source Code:** You can view the CSRF token in a browser by Right Click -> **View Page Source** and looking for `<input type="hidden" name="_csrf" value="...">`.
